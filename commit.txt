Directory input (submission or context), and order-insensitive flags

Two things.

1. Order-insensitive flags. The positional `rest` was `trailing_var_arg +
   allow_hyphen_values`, so any flag placed AFTER the question text was
   swallowed into the positional as literal words (`ask "q" --eagles 5`
   never parsed --eagles). Dropped both attributes; clap now interleaves
   options and positional words freely — flags work before, after, or
   between the words.

2. Directory input. New read_path reads a FILE or a DIRECTORY: a directory
   is walked (sorted, recursive, skipping .git/target/node_modules/.lake/…
   and binary extensions, capped at 2M chars) and its text files
   concatenated under `===== relative/path =====` headers. Wired into
   --file, --ask, and a bare positional path, so a whole corpus can be the
   submission in one shot. New --context PATH (repeatable, file or dir)
   loads a corpus as BACKGROUND, prepended under a CONTEXT header ahead of
   the SUBMISSION — the "as context" vs "as the submission" distinction.

Verified live: dir as submission, --context dir + question, --file dir,
and flags after/between positional words all resolve correctly. read_path
unit-tested (file, dir concat with headers, binary excluded). Removed the
now-dead content_len_label. Suite green (32 passed).
