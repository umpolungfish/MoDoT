Surface what the API actually said, and stop --no-think killing a run

A jam died on `status code 400` and the abort blamed credit: "not transient:
402 = out of credits, 401 = bad key". Both wrong, and undiagnosable by
construction — ureq::Error::Status(code, resp) stringifies to a bare status line
and DROPS the response, so the server's answer was sitting inside the error and we
printed the number. is_fatal_llm_error did not know 400 either, so the loop kept
re-sending a request that would refuse forever.

Read the body. It said exactly what was wrong, first time:
  {"error":{"message":"Reasoning is mandatory for this endpoint and cannot be
   disabled.","code":400}}

It was --no-think. google/gemini-3.5-flash mandates reasoning and rejects the
request over `reasoning: {enabled:false}`. Nothing to do with money.

My own note on that switch said it was a "no-op on models that don't support it".
That is false in both directions and I had already seen the other half today:
DeepSeek's native API 400s on the field's mere PRESENCE (so it stays
OpenRouter-only), and this endpoint 400s on its VALUE. The switch is not harmless;
it is a request some endpoints refuse.

--no-think is an OPTIMISATION and must never be able to end a run. Where the
endpoint mandates reasoning we now drop the switch and proceed, saying so. 400
also joins the fatal set: a malformed request will not un-malform on retry, so
abort with the body rather than grind.

Verified against the model from the failing run: gemini-3.5-flash now prints
"this endpoint mandates reasoning — dropping --no-think" and completes the jam.
gemini-3-flash-preview accepts the switch; the run used both, which is why one arm
worked and one killed it.
