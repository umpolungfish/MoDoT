Port the kernel-gated prover into ask_native (native, no Python)

`./ask` runs the Rust ask_native binary, so the type-router and prover I built in
the Python package were unreachable from it. A proof question came back as a
scaffold with a `sorry` in it. This ports the prover into ask_native itself, in
Rust, so `./ask` can close a goal through the Lean kernel.

src/prover.rs is the closed loop: generate a Lean tactic block, write it into the
p4ramill scratch module, run `lake build`, read the unsolved-goal state the
compiler prints, and repair against it until the build is green with no sorry.
When a goal will not close flat it is FSPLIT into helper lemmas, each proved
recursively and FFUSEd back, with memoization by statement so a lemma proved once
is reused and a header dedup so a reused lemma collapses to one declaration. It
shells out to `lake`, the Lean toolchain, a separate binary. Nothing here touches
Python.

Not closed is a navigation frontier, never a verdict of unprovability. The driver
escalates depth and budget across rounds and keeps decomposing; the only stop is a
resource cap, which is a statement about compute, not about the theorem. A
provable result closes given the work.

Wired into run_one: a `prove:` prefix or a literal Lean theorem or lemma routes to
the prover instead of the prose spine. Everything else flows the normal
ManuscriptSpine unchanged.

Verified: `./ask --ask "prove: theorem ask_native_ok : (2 : Nat) + 2 = 4"` routes
to the prover and closes green through the real Lean kernel with rfl, no sorry, no
Python. The scratch module is restored to its green placeholder after the run.
