Stop the ascend/imscribe runaway: refuse decorated name-ladders

A cycle went runaway: the agent iterated `ascend` on K19 and imscribed
each "tower level" — K19⁺, K19⁺⁺, K19⁺⁺⁺, … — 87 calls in one round,
registering a ladder of near-duplicate catalog entries. A derived state
(excited / ascended / radical) is a TRANSFORM of its base, not a new
catalog entity, so enumerating levels into the catalog is both pollution
and non-terminating.

run_imscribe now has a decoration-ladder guard: if a name is some base
plus a trailing run of decoration marks (⁺ • * + † …) and that base
already exists (case-insensitively), it refuses to register and tells the
agent the derived state is a transform of its base — the verb that built
it already returned the tuple, and a tower that only climbs by
re-imscription is a B frontier, not a catalog to enumerate.
