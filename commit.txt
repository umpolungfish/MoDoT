Add a DeepSeek provider; abort on fatal LLM errors; warn on unknown provider

Three problems surfaced when OpenRouter ran out of credits mid-jam:

1. The 402 flood was OpenRouter returning Payment Required. Nothing wrong
   in the code, the account was empty.

2. `--provider deepseek` silently did nothing. parse_provider only knew
   openrouter and gemini, so "deepseek" returned None and the resolver
   fell through to key-based selection, which picked OpenRouter (the key
   was set). The header even said "(openrouter)" while the flag said
   deepseek. The run never left the dead provider.

3. The jam ground through all 40 cycles on a 402, each cycle re-calling
   and failing identically, because a fatal auth/payment error was treated
   like a transient one.

Fixes:
  - Provider::DeepSeek: base_url api.deepseek.com/v1, key DEEPSEEK_API_KEY
    (then MODOT_API_KEY). DeepSeek's API is OpenAI-compatible, so it reuses
    the OpenRouter inference path verbatim. parse_provider accepts
    deepseek / ds / deepseek-direct, and key-based inference now considers
    DEEPSEEK_API_KEY.
  - An unrecognized explicit --provider / MODOT_PROVIDER value now prints a
    warning naming the valid providers instead of silently falling back.
  - is_fatal_llm_error: a 401/402/403/invalid-key/insufficient error aborts
    the cycle loop immediately with a clear message (switch providers or top
    up), rather than retrying every cycle.

Verified: `--provider deepseek --model deepseek-chat` routes to DeepSeek
(header shows "(deepseek)", a live call returns), and an OpenRouter 402
aborts the run after cycle 1 instead of grinding all of them.

Note: DeepSeek's API model ids are deepseek-chat and deepseek-reasoner;
deepseek-v4-pro is not one, so pass a real id when using --provider deepseek.
