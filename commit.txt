arrange now reports closure MULTIPLICITY -- kills the "only sequence" overclaim

Watching a "design a computer from {grothendieck_topos, general_recursive_function,
skolem_normal_form}" run, the agent declared its best order "the only configuration that
achieves closure" -- and its own close call had just cyclized a DIFFERENT order. The
overclaim traces to arrange: it searched all 6 orderings but printed only the single best,
so a reader (or the model) reads "best" as "only."

Fix in ask_native/src/click.rs: best_ordering now returns an Arrangement struct that also
tallies how many searched orderings fully enchain (n_full) and how many cyclize (n_cyclic).
run_arrange reports it: closure is order-ROBUST ("N/M searched orderings cyclize ... NOT the
only sequence that closes -- do not call it unique") vs order-SPECIFIC ("the only ordering
that cyclizes") when exactly one closes. Verified: the exact set now prints "order-ROBUST:
4/6 searched orderings cyclize," so the uniqueness claim can no longer be confabulated.
