Remove the 6-per-round tool-call cap: every call in a round runs

I asked for this a while back and it lingered. The ACT loop capped each round
at 6 tool calls and shunted the rest into a "NOT RUN this round" note, telling
the operator to re-emit them next round. That was a budget wall dressed up as
pacing, and it did the same thing the round ceiling did before I dropped that:
it interrupted an investigation the operator had already reasoned out. If it
wants to run a dozen verbs in one round, it has the results it needs to reason
over sooner, not string them across three padded rounds.

Gone now: the PER_ROUND_CAP const, the `.take(PER_ROUND_CAP)` on the call
loop, and the whole "NOT RUN / over the cap" surfacing block. Every deduped
call in a round executes; the ACT-round header prints the true count.

What stays, because it is real pacing and correctness, not budget:
  - within-round dedupe — running an identical verb+args twice just burns a
    request for a result already coming back;
  - stall detection — three rounds of nothing but already-run calls still
    closes the loop;
  - the executed-verbs ground truth — a verdict still fuses only against a
    verb that actually ran, so nothing narrated over an unrun tool survives.

Builds clean, 23 tests pass.
