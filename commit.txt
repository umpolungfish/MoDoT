Make the operator ACT in a loop instead of narrating -- THINK/ACT/OBSERVE/UPDATE

Lando asked why the operator keeps telling him what it needs to do instead of just
doing it. The honest answer is that I built it single-shot: one draft, run at most 6
tools decided all at once in parallel, one synthesis. No loop. So when a task needs
"do X, see the result, then do Y based on it" -- polymerize, observe the termination,
close, use the linker it found, re-polymerize, characterize -- the operator could not
DO the follow-ups, so it NARRATED them as prose ("Case A ... the optimal path is ...").
It described the steps because the architecture gave it no next step.

Replaced the single-shot tool block in ask_native/src/main.rs with a real agentic loop
(THINK->ACT->OBSERVE->UPDATE):
- the operator emits TOOL: lines (ACT); the harness runs them over the real catalog
- the outputs come back as ground truth (OBSERVE), still under the golem constraint
  (never contradict a tool, introduce nothing it did not return)
- the model reads them and chooses the NEXT act (UPDATE), iterating up to MAX_ROUNDS,
  then answers with no TOOL: lines
- if it hits the step cap mid-action, one forced clean grounded close
The final looped answer (not the draft) now feeds the ManuscriptSpine and the multi-turn
history. New loop system prompt: "run a tool, read its result, run the next ... NEVER
narrate a step you could run; run it."

Verified live on a dependent chain -- combine {binah monad ankh zosimos_alchemy
sefer_yetzirah}, and if it terminates find a linker, insert it, re-run, then characterize:
- round 1: polymerize -> termination at ankh-zosimos; close -> linker plutonium_normal
- round 2: it INSERTED plutonium_normal and re-ran -> DP 3 -> 6 units, repair confirmed
- round 3: close on the new 6-chain
It used the linker from round 1's close inside round 2's re-polymerization -- genuine
iteration on real output, impossible in the old single-shot flow. It did the steps
instead of describing them.
