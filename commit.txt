I gave the excriber the primitives as tangible handles.

IMASM names a program's characteristics abstractly (fork, fuse, work, hold);
the twelve primitive axes give those characteristics grounded names an object
can be judged against. Three groundings now ride the excriber's prompt:

- Each opcode in the alphabet is tagged with the primitive axis it rides, from
  the canonical IGProtocol field assignment: VINIT Dimensionality, TANCH
  Topology, AFWD Relational, AREV Polarity, CLINK Fidelity, IMSCRIB Interaction
  Grammar, FSPLIT Granularity, FFUSE and ENGAGR Stoichiometry, EVALT
  Criticality, EVALF Chirality, IFIX Topological Protection (Kinetic Character,
  the twelfth, is carried by the register's motion, not one opcode). The tri
  face tags FSPLIT3/FFUSE3/EVALI/TNEG/INEG the same way.
- The twelve axes themselves, each with a one-line tangible handle.
- The primitive types loaded from ob3ect/digital (the_primitive_type_called_*),
  each rendered as name and its own glyph word: 50 concrete exemplars of how the
  characteristics compose into real programs, cached on first use, dropped from
  the hotter retry to leave room to diverge.

The excriber is told to read what the program's shape says about each axis and
name the real object with those characteristics. The em-dashes in the alphabet
table are replaced with colons in the same pass. Guide Part XIV notes the
grounding.

Verified: build green under local,cuda; 7/7 learn tests; 50/50 primitive type
ob3ects parse to glyph words; a live run shows the grounded excriber producing
domain-varied guesses (garden techniques, spherical coordinates, cell division).
