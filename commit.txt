Strip set punctuation from agent tool args -- braces were leaking in as bad monomer names

My own TOOLS_PROMPT tells the operator that a `{set}` in braces is unordered and to reach
for `arrange`. So the model does exactly that: it emits `TOOL: arrange {binah monad ankh}`
-- braces and all -- because that is how the set was written in my question. The harness
then shelled those tokens straight through as monomer names, and `arrange` died on the
first one: "monomer not found: {binah". `close` and `material` only survived because the
model happened to strip the braces for them; `arrange` got the raw set.

That is the same silent-arg-corruption class as the earlier click arg-drop: set notation
is not part of any entry name, and the model should not have to hand-clean it. The fix is
one spot -- the choke point in run_structural_tool (ask_native/src/main.rs) where an
emitted verb becomes a real subprocess. Before dispatching, normalize each arg: trim, then
strip surrounding set punctuation ({ } [ ] ( ) ,) from both ends and drop any token that
was pure punctuation. `+` is deliberately left alone -- it is the pre-click token, and
`monad+binah` must stay a single blended monomer. Every verb now sees clean names, and the
click arity guard counts real names instead of `{binah`.

Verified live through the agent path, not just the CLI: asking the same {binah monad ankh}
set, the operator emits `TOOL: arrange {binah monad ankh}` and it now RUNS -- "searched 6
ordering(s) ... best ordering FULLY enchains all 3 units AND CYCLIZES into a ring" --
instead of erroring on the brace. The ACT/OBSERVE loop closed clean: fused=T, conflict=0.
