Make catalog load survive a mid-write truncated read

The cache-clear landed — the agent now runs filter / ascend /
phase_reconstruct and discards the stale "unavailable" claim. But the
same cycle exposed a race: `material` failed with "catalog json: EOF at
line 1 column 0 / no catalog loaded" while `forge` a moment earlier
loaded the catalog fine. The catalog file is rewritten in place by other
processes (imscribe, the live updater); a shelled verb that reads it
mid-write sees an empty/truncated file and spuriously drops a real result
to N.

load_catalog now retries the read+parse up to six times with a short
backoff, so a concurrent write can finish before we give up. An
empty/truncated read is treated as transient; a genuinely malformed
catalog persists across the retries and still errors honestly. Parsing is
split into load_catalog_value so the retry wraps only the volatile
read+decode. Verified `material` now loads and forges (ρ=2.0000); suite
green (29 passed).
