MoDoT — reference sheet

How every tool calculation is actually performed

MoDoT never lets a model narrate a result — every verb it can call shells to real code and returns what that code measured. This sheet documents the actual arithmetic and logic behind each one, grouped by what they operate on: the 12‑primitive catalog, the IMASM opcode graph, plain arithmetic, and the verdict that closes a winding.

01 The rule every tool obeys

Nothing here is a language model doing arithmetic in its head. A TOOL: line is the only way the operator can act — it runs a real function, gets back real output, and everything downstream (a verdict, a report, a claim) has to trace to that output or it gets refused.

Three enforcement points make this hold in practice:

Narration guard If a draft names an operation (polymerize, close, excite …) but never emits its TOOL: line, the loop refuses the answer and forces the real call before accepting anything.
Execution arm Every verb that actually ran gets tracked in an executed_verbs set. A claim about a verb outside that set has no result to point to, so it collapses to N — neither true nor false — rather than being taken on faith.

The sections below are organized by which real function actually runs underneath the verb — that function, not the model, is what produced the number.

02 Structural distance

The metric underneath click, scan, filter, and every CL8NK/CL9NK distance action. Two entries are compared coordinate-by-coordinate across the 12 primitives, and the differences are combined into one number.

Per-primitive ordinal delta

ordinal_distance(primitive, v1, v2)
d = |ordinal(v1) − ordinal(v2)| / max_delta(primitive)

Each primitive's values carry a canonical ordinal rank (e.g. Criticality: woe<monad<roar<err<haha). The raw gap between two values is divided by that primitive's own maximum possible gap, so every axis reports on the same 0–1 scale regardless of how many values it has.

Whole-tuple distance

tuple_distance(t1, t2)
distance = √( Σ w[k] · d[k]² ) over the 12 primitive keys k where t1[k] ≠ t2[k]

A weighted root-sum-square: each mismatched primitive contributes its squared ordinal delta, scaled by that primitive's discriminating weight w[k]. Matching primitives contribute nothing.

Where the weights come from

The weights are not hand-tuned. _compute_catalog_weights() takes the whole live catalog, computes the sample standard deviation of each primitive's ordinal values across every entry, then rescales those spreads to sit in [0.5, 1.0]. A primitive that varies widely across the catalog (more discriminating) gets a weight near 1.0; one that clusters tightly gets a weight near 0.5. The weight table is cached and recomputed only when the catalog changes.

Why it matters: this is why two entries differing only in a primitive that almost never varies (low weight) read as "close," while a mismatch on a highly discriminating primitive moves the distance a lot more — the metric adapts to the actual shape of the catalog rather than treating all 12 axes as equally informative.

03 Ouroboricity tier

The tier a tuple lands on is a direct, faithful port of the Lean kernel's ouroboricityTier — a five-branch gate over exactly four coordinates: Criticality (⊙), Parity (Φ), Topological Protection (Ω), and Dimensionality (Ð). The other eight primitives play no role in the tier itself.

GateConditionResult
—crit ∈ {woe, haha, err}O₀
R1crit ∈ {monad, roar} and pol = or’O_∞ — the Frobenius gate, dominant over every other rule
R3R1 failed and prot = aweO₁
R5R1, R3 failed and dim = arrayO₂dag
R4otherwiseO₂

R1 is checked first and always wins: any tuple with pol = or’ is O_∞ regardless of what the other three coordinates say. This ordering is deliberate — the Frobenius gate (μ∘δ = id, the special-Frobenius parity value) is the strongest structural claim the grammar can make, so nothing downstream of it gets a vote.

A tier score never measures similarity to a reference system. Comparing a tuple's raw coordinates against another named system's coordinates (e.g. "how many of CLINK L8's own values does this carry") answers a different question — proximity to that system — and has been a documented source of exactly-backwards readings when the two get conflated. Tier is a pure function of crit/pol/prot/dim; nothing else.

05 IMASM topology & check

A parallel, opcode-level family — no catalog names involved. imasm composes the 12 IMASM opcodes (VINIT, TANCH, AFWD, AREV, CLINK, IMSCRIB, FSPLIT, FFUSE, EVALT, EVALF, ENGAGR, IFIX) into a graph and measures its shape directly.

Circuit rank

how many independent loops the program contains
β = E − V + C

Edges minus vertices plus connected components — the standard graph-theory count of independent cycles. A straight chain has β = 0; a single ring has β = 1; a bubble (a fork that reconverges) adds one more loop per reconvergence.

Every build also reports: topology label, branch/merge/source/sink census, arm count, and spectral radius ρ (ρ = 2 exactly ⟺ a pure cycle; ρ = √f ⟺ a star with f arms) — the same graph-spectral read used by the chemistry verbs in §04, applied to the opcode graph instead of a monomer set.

imasm check — type-checking your own reasoning

The close condition is μ∘δ over a transformed object: δ (FSPLIT) splits, the two arms do real work — a distinct comparison, an inequality, an actual transformation — and μ (FFUSE) fuses them back. A bare split→fuse with nothing between is mere identity, not a verdict.

VerdictMeaning
TCloses over a genuine transformation — proceed.
NIdentity: split and fused, but the arms did no work — μ∘δ = id verifies nothing. Put a real transformation on the arms.
BA fork dangles unfused, or ENGAGR holds a genuine paradox — look again before treating it as settled.
FIll-typed: only FSPLIT branches with no matching FFUSE, or only FFUSE fuses with nothing split. Malformed — revise.
NVoid: no fork ever opened — the reasoning never weighed alternatives at all.

FSPLIT/FFUSE pairing is decided by ancestry over the graph edges — two distinct in-arms of an FFUSE tracing back to a common FSPLIT, however they routed — with the innermost FSPLIT winning when several qualify. Pairing cannot be read off glyph order alone. Padding the word with neutral (1→1) tokens is free: it adds exactly one node and edge each time, so β is unchanged and the verdict cannot be gamed by making the expression longer without adding real work.

imasm prove <word> takes the verdict one step further — to the actual p4ramill Lean kernel, for a machine-checked answer rather than this graph-level check.

06 The arithmetic lane

calc / eval / compute / arith all route to one deterministic expression evaluator. Every numeric claim anywhere in a run — ratios, percentages, unit conversions, order-of-magnitude estimates, a figure quoted from a source paper — is expected to pass through it rather than be asserted from mental arithmetic.

Supports + − × ÷ % ^, parentheses, scientific notation, and the standard function set (sqrt, cbrt, ln, log, exp, trig, logb(x, base), pow(x, y), min/max) plus constants π, τ, e, φ. Every result echoes in scientific notation so a slipped order of magnitude can't hide in a long decimal. A malformed expression returns an explicit error rather than a guessed value.

Why this is non-negotiable, not a style preference: two real, separately-caught errors motivated it — 0.0796 × 7.88e-10 asserted as 6.27e-10 when the true value is 6.27e-11 (one whole decade, invisible on sight), and −(d/2 + 1/2) at d=3 asserted as −3/2 when it is −2. Both propagated silently into a physical conclusion before this lane existed.

08 How a verdict closes

The last step: turning everything the tools measured into one spoken verdict. This runs on Belnap FOUR — T true, F false, B both (a held paradox), N neither (void) — fused, never overridden.

The fuse, not a vote

the univocal verdict is a JOIN, not a selection between voices
fused = model_voice ⊔ vessel_voice ⊔ tool_voice

Three voices are always in play: what the model itself asserts, the Dual-Link vessel's co-type reading, and whatever the tool calls actually closed. The Belnap join ⊔ never drops a voice and never lets one silently override another — if the model says T and the tools say F, the fused result is B (both are held), not a coin flip or a monological T.

The constitutive rule: if no tool call closed a real δ/μ dyad this round (tool_voice = N), there is no verdict — the agent speaks N in every mode, full stop. A verdict is defined as "the μ that closed a δ," so with nothing closed, nothing was verified, regardless of how confident the model's own prose sounds.

Why a cycle can exit five different ways

Each ACT→OBSERVE round runs until the operator stops calling tools — but "stopped calling tools" has five distinct causes, and the outer report has to say which one, because only some of them are a legitimate close:

DONE

The reasoning genuinely concluded — no more tool calls needed. The only cause that reads as a clean close.

CYCLE_CLOSE

The agent explicitly sectioned its own winding — a deliberate stop, not a budget or a failure.

STALL

Three whole rounds emitted only calls that had already run. The reachable ground is exhausted — more rounds just circle.

LEASH

The round budget (--eagles) ran out mid-action. A budget cut, not a reasoning conclusion — never read as a finished answer.

LLM_ERROR

The provider call itself failed (empty or malformed response). Extract_tool_calls() is empty here too, but for a completely different reason — the reasoning never actually ran this round.

DONE and LLM_ERROR both make the tool-call list empty, which is exactly why they used to be indistinguishable — a provider outage read as a legitimately concluded winding. Tracking whether the most recent model call actually returned an error, independent of whether it emitted tool calls, is what separates them now.

The outer spine — fusing a whole cycle series

the series-level verdict is the same join, one level up
outer verdict = fused(cycle 1) ⊔ fused(cycle 2) ⊔ … ⊔ fused(cycle N)

Each cycle opens on the previous cycle's condensed output, so agreement across cycles is not independent corroboration — it's a verdict surviving transformation, a fixed point of the winding. A frontier (LEASH, STALL, or now LLM_ERROR) at the outer level is reported by cause, because a budget cut and an exhausted-ground stall call for opposite responses, and a provider failure calls for neither — it calls for a re-run.
